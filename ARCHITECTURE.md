# ARCHITECTURE

## 1. Цель системы
Система переводит работу с силлабусами из файлового режима (Word/PDF + почта) в управляемый онлайн-процесс:

1. Единая база курсов и тем.
2. Быстрая сборка силлабуса на семестр.
3. Автоматическая ИИ-проверка.
4. Прозрачное согласование по ролям.
5. История версий и действий в одной БД.

## 2. Роли и ответственность

1. `teacher` / `program_leader`:
   создание курсов, банка тем, сборка/импорт силлабуса, отправка на проверку.
2. `dean`:
   согласование после ИИ, возврат на доработку при замечаниях.
3. `umu`:
   финальное утверждение или возврат на доработку.
4. `admin`:
   административный доступ и поддержка процесса.

## 3. Ключевые доменные сущности

1. `Course`:
   дисциплина преподавателя (код, названия, языки, общность).
2. `Topic`:
   тема курса (часы, описание, тип занятия, активность).
3. `TopicLiterature`, `TopicQuestion`:
   литература и вопросы к теме.
4. `Syllabus`:
   документ на конкретный семестр и учебный год.
5. `SyllabusTopic`:
   какие темы из банка вошли в конкретный силлабус.
6. `SyllabusRevision`, `SyllabusStatusLog`, `SyllabusAuditLog`:
   история изменений, переходов и действий пользователей/системы.

## 4. Основные сценарии работы преподавателя

### Сценарий A: Конструктор
1. Создать/обновить курс и банк тем.
2. Создать силлабус.
3. Выбрать темы из банка, настроить недели/часы.
4. Заполнить детали силлабуса.
5. Отправить на проверку ИИ.

### Сценарий B: Импорт готового файла
1. Создать силлабус.
2. Загрузить PDF/Word.
3. Отправить на проверку ИИ (или автостарт для upload-flow).

Важно: это альтернативные сценарии. Достаточно одного.

## 5. Workflow статусов

Базовая цепочка:

1. `draft`
2. `ai_check`
3. `review_dean`
4. `review_umu`
5. `approved`

Ветка возврата:

1. `correction` (с комментариями)
2. повторная отправка в `ai_check`

Архивный статус:

1. `rejected`

## 6. Архитектура ИИ-проверки

Пайплайн:

1. Извлечение текста из файла (PDF/Word) или сборка текста из БД.
2. Быстрые правила (`fast-rules`) для типовых решений.
3. Защита от нерелевантных файлов (`not-syllabus detection`).
4. LLM-проверка для сложных случаев.
5. Сохранение результата, комментариев и итогового статуса.

ИИ в проекте — инспектор качества, не генератор силлабуса.

## 7. Почему архитектура подходит для диплома

1. Решает исходную прикладную проблему вуза.
2. Имеет четкую доменную модель и ролевую логику.
3. Покрывает полный жизненный цикл документа.
4. Обеспечивает трассируемость и управляемость версий.
5. Масштабируется на другие кафедры и дисциплины.
